### gas/Code.gs (발송 워커 + 학습 + autoSend)
- `ingestMessage_`: **threadId 저장**(원 스레드 식별자). `makeDraftFor_`: draft에 threadId·origMsg 전파.
- `sendApprovedDrafts()`(pollCsInbox run 내, 수신과 분리): approved → **원 스레드 `reply`**(새 메일 작성 금지) → sent+sentAt / 실패 error+errorMsg. **원자적 잠금**: 최신 재확인 → `sending`+sendingAt → 재조회 확인 후 발송. **sending 10분+ 스턱 → error 자동 전환.** threadId 없으면 발송 안 하고 error. Gmail 발송도 `budgetGate_` 경유.
- `autoApprovePass_()`: 초안 생성 직후 `cs/config/autoSend`=true & **confidence ≥ 0.8**이면 자동 approved(editedByClara:false, autoApproved:true) → 같은 run 발송. **<0.8 또는 null은 pending 유지(안전핀).**
- 학습 루프(발송 성공 시): 무수정 → `cs/corpus {origin:'approved'}`(정답 적재) / 수정 → `cs/learn {before,after,orig,lang,ts}`(쌍 저장).
- 순수 판정 헬퍼: `sendDecision_`·`autoApproveDecision_`·`learnTarget_`.

## 리허설 절차 (클라라, 실코드 단일 경로)
1. **gas/Code.gs 재반영**(PWR-CS-Engine, paradisewalkresidence 계정). 트리거는 `installCsTriggers()`로 5분 1개 유지.
2. **가짜 게스트 메일**: joi.hurricane → **paradisewalkresidence@gmail.com**. 파서가 인식하도록 **부킹 알림 형식**으로 본문 작성:
   - 제목: `테스트게스트 님의 메시지가 도착했습니다`
   - 본문에 포함: `예약 번호: 1234567890` / `테스트게스트 님의 메시지:` / (질문 한 줄, 예 `What time is check-in?`) / `답변 -->`
   - 그 메일에 **`cs/booking` 라벨** 부여(필터 또는 수동).
3. 5분 대기(또는 에디터에서 `pollCsInbox` 1회 실행) → `cs/inbox` 적재(threadId 포함) → 초안 생성 → 승인앱 **대기 탭에 카드** 표시.
4. **수동 발송 테스트**: 앱에서 (선택) [수정] 후 [승인·발송] → "발송할까요?" 확인 → 카드 approved(스피너). 다음 run(≤5분 또는 수동 실행)에 GAS가 **원 스레드에 회신** → **joi.hurricane 수신함에 답변 도착** 확인, 카드는 **보냄 탭(sent)**로 이동. 학습 확인: 무수정→`cs/corpus`(origin approved) / 수정→`cs/learn`.
5. **autoSend 테스트**: 앱에서 토글 ON → 새 가짜 메일 1건. 모델 confidence가 **0.8 이상이면 승인 없이 자동 발송**(보냄 탭에 `자동` 배지), 0.8 미만이면 **대기 유지**(안전핀). 어느 쪽인지 로그/화면으로 확인.
6. **재시도**: 만약 error가 나면(예: threadId 없는 구 초안) 카드의 [재시도] → approved로 되돌아가 다음 run 재발송.

## 위험 잔존 사항
- **이중발송 원자성**: RTDB REST에 트랜잭션이 없어 "sending 잠금 + 재조회" 방식(지시문 명시). **단일 5분 트리거라 run 겹침이 사실상 없어 안전** — 단, 트리거를 여러 개 만들면 위험. `installCsTriggers()`가 항상 1개만 설치하도록 보장하니 **수동으로 추가 트리거를 만들지 말 것.**
- **autoSend는 모델 confidence에 의존**: 모델이 과신해 0.8+를 주면 자동 발송됨. **초기엔 autoSend OFF로 두고 무수정 승인율을 관찰한 뒤 켤 것**(설계 원칙과 동일).
- **reply 대상**: 스레드의 마지막 메시지 발신자에게 감(정상 부킹 스레드=게스트). 발송 후 게스트가 또 보내면 새 pending 생성(정상 루프).
- **구 초안(threadId 없음)**: M2b-2 이전 적재분은 발송 시 error 처리 — 정상(재수신 필요).
- **Firebase 규칙**: 인증 사용자에게 `cs/drafts` 쓰기 + `cs/config` 쓰기 허용 필요(M2b-1과 동일 전제). `cs/learn`·`cs/corpus`는 GAS(FB_AUTH)만 쓰므로 규칙 무관. 승인 시 "쓰기 권한" 토스트가 뜨면 규칙에 `cs/drafts`·`cs/config` 쓰기 추가 필요.
